--------------------------------------------------------------------------------
  1. WHAT WAS CONSOLIDATED
--------------------------------------------------------------------------------

  Three previously loose capabilities were unified into a single, reproducible
  text-to-speech workflow with version-controlled deployment:

    (A) Media-agnostic text extraction
          - LaTeX manuscripts      (\input, \cite, \ref, math, macros)
          - PDF papers             (pdftotext + boilerplate stripping)
          - Markdown documents     (headers, bold/italic, tables, code fences)
          - Plain-text reviews     (bullets, rules, pipe tables, headers)
          - URLs and office docs   (via pandoc pre-conversion)

    (B) Voice synthesis + hallucination screening
          - Coqui TTS VCTK VITS (voice p246, deep male, 0.90x speed)
          - Pre-synthesis text screening (10 pattern categories)
          - Post-synthesis audio screening
          - Automatic chunk cleanup, final MP3 only

    (C) One-command invocation from any Claude Code session
          - Slash command `/speak` that auto-dispatches by media type
          - Deterministic output contract
          - Installed via symlinks so `git pull` propagates updates

--------------------------------------------------------------------------------
  4. THE CLEANER TAXONOMY
--------------------------------------------------------------------------------

  All cleaners ultimately emit a "TTS-ready" plain text string. They share a
  common helper, _expand_for_tts(), which handles symbol/unit/math expansion
  (rho -> "rho", |rho| -> "absolute rho", mug -> "micrograms", etc.).

    LaTeXTTSCleaner
      - Strips \begin{figure}, \begin{table}, \begin{equation*} blocks
      - Inlines \cite{...} as "citation" and removes reference clutter
      - Expands math: \rho, \mu, \sigma, \sum, \int
      - Resolves \input{} recursively
      - Discards comments, \label, \ref bodies, \footnote, etc.

    PDFTTSCleaner
      - Runs pdftotext -layout
      - Strips running headers, footers, page numbers
      - Collapses broken word-wraps at line boundaries
      - Feeds the result through _expand_for_tts()

    PlainTextTTSCleaner
      - Strips horizontal rules (---, ===)
      - Skips pipe-delimited tables
      - Strips bullet markers: dot, bullet, dash
      - Converts "REVIEWER N - Field" -> "Reviewer N, Field."
      - Converts "Major Issues:" headers -> "Major Issues."
      - Removes (line 123) and (lines 456-789) references

    MarkdownTTSCleaner
      - Converts # headers to spoken prose
      - Strips **bold**, *italic*, `code`
      - Removes (commit abc123) parentheticals
      - Removes file:line references (main.tex:187)
      - Converts "Status: ADDRESSED" -> "Status: Addressed"
      - Skips pipe tables and horizontal rules

    clean_multi_files()
      - Dispatches each input to the right cleaner by extension
      - Prepends spoken section headers (user-supplied or auto-generated)
      - Joins with '\n\n\n' dividers
      - Returns concatenated text + per-file stats

--------------------------------------------------------------------------------
  5. THE HALLUCINATION SCREENER
--------------------------------------------------------------------------------

  TTSHallucinationScreener analyzes text BEFORE synthesis and (optionally)
  audio AFTER synthesis. Ten pattern categories are tracked:

    LONG_NUMBER        digits >= 6 in a row (e.g. job IDs, PMIDs)
    WORD_REPEAT        same token repeated 3+ times consecutively
    BARE_YEAR          four-digit year without context
    URL                http/https embedded in flow text
    EMAIL              name@host.tld
    HEX                git short hashes, RGB codes
    ALL_CAPS_RUN       ALL-CAPS words >= 3 in a row (acronym storms)
    PUNCT_RUN          punctuation >= 4 in a row
    NONWORD_RUN        pattern of non-alphanumeric gibberish
    EQUATION_REMNANT   stray LaTeX/math that slipped past a cleaner

  Each trigger is recorded with location + excerpt in
  hallucination_report.json, alongside audio-side findings if any.

--------------------------------------------------------------------------------
  11. DESIGN DECISIONS AND TRADE-OFFS
--------------------------------------------------------------------------------

  Why symlinks for the slash commands?
    A single git pull in ~/Documents/claude-skills/ updates every installed
    command atomically. No rsync, no stale copies, no drift between "what
    the repo says" and "what Claude Code sees". The cost is one level of
    indirection, which Claude Code handles transparently.

  Why file-level symlinks, not a directory symlink?
    ~/.claude/commands/ contains at least one local-only helper file
    (export_session.py, used by /log). Symlinking the whole directory into
    the repo would either clobber the helper or force it into the repo.
    File-level links preserve the mix.

  Why a copy of media_to_tts.py in both /media/drn2/External and
  ~/Documents/working-tts-gpu/gpu-tts-toolkit?
    The External drive is the runtime location where the conda env, cached
    TTS model, and input data all live. The repo checkout under
    ~/Documents/ is the version-controlled source of truth. During active
    development they can temporarily diverge; a `cp` + commit + push brings
    them back into sync.

  Why voice p246 as default?
    Deep male VCTK speaker with strong prosody, benchmarked on this
    machine for lowest hallucination rate on scientific text at 0.90x
    speed. Overridable per invocation via --voice.

  Why always screen?
    Hallucination detection is cheap relative to synthesis, and silent
    failures (repeated phrases, garbled numbers) are the most common
    failure mode of unscreened TTS. Always-on screening + structured
    report means every run is auditable.
